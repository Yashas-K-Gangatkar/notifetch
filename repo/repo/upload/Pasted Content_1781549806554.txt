## PROJECT CONTEXT

**App:** NotiFetch — Android app that captures delivery platform notifications
using NotificationListenerService, parses them for order values/locations/
distances/categories, stores them in Room, and syncs to a backend.

**Two user modes:**
- Rider/Driver mode: gig workers on Swiggy, Zomato, Uber Driver, Blinkit, etc.
- Customer mode: online shoppers on Swiggy, Zomato, Amazon, Flipkart, etc.

**Product philosophy:** Trust-first, revenue-free. All paywalls removed in v2.8.0.
Global adoption over monetization.

**Key facts:**
- Package: com.notifetch.app
- Current version: 2.8.1 (versionCode 25)
- Min SDK 24, Target SDK 35 (Android 7.0 → Android 15)
- Language: 100% Kotlin + Jetpack Compose + Material3
- Architecture: MVVM + Hilt + Room v5 + Retrofit + WorkManager
- Backend: https://www.notifetch.in/api/ (Next.js on Vercel)
- Firebase: Auth (Google Sign-In), FCM, Crashlytics, Analytics
- Room schema: v5, two tables (captured_notifications, platform_configs)
- Platform coverage: 55+ rider packages + 40+ customer packages (95+ total)
- Repo: https://github.com/Yashas-K-Gangatkar/d2

**Gradle/build:**
- AGP 8.7.3, Kotlin 2.0.21
- Compose BOM 2024.12.01
- Hilt 2.53.1, Room 2.6.1, Retrofit 2.11.0, Firebase BOM 33.7.0

### BUG 1: runBlocking on main thread (ANR risk)
**File:** `notifetch-android/app/src/main/java/com/notifetch/app/MainActivity.kt`
**Lines:** approximately 46-50

**Problem:** DataStore preferences are read with `runBlocking` before `setContent`.
This blocks the main thread and causes ANR on cold starts or slow devices.

**Fix:** Remove the `runBlocking` block. Use a `SplashScreen` or a `var` with a
default value, then update reactively in a `LaunchedEffect`.

**Pattern to use:**
```kotlin
// BEFORE (wrong — blocks main thread)
val isDarkMode = runBlocking { dataStore.data.first()[DARK_MODE_KEY] ?: false }

// AFTER (correct — async, default then update)
@Composable
fun MainContent() {
    var isDarkMode by remember { mutableStateOf(false) }
    LaunchedEffect(Unit) {
        dataStore.data.collect { prefs ->
            isDarkMode = prefs[DARK_MODE_KEY] ?: false
        }
    }
    NotiFetchTheme(darkTheme = isDarkMode) {
        // nav host here
    }
}
```

### BUG 3: extrasJson stores entire notification Bundle (CRITICAL PRIVACY)
**Files:**
- `data/local/CapturedNotification.kt` (the Room entity)
- `notification/NotiFetchListenerService.kt` (where it's populated)
- `util/Helpers.kt` (the `extrasToJson()` function)

**Problem:** `extrasJson = Helpers.extrasToJson(extras)` stores the entire
notification Bundle. This Bundle contains data the user never sees:
customer PII (names, phone numbers), auth tokens, deep-link URIs. This violates
GDPR Art. 5(1)(c) and India's DPDPA Rules 2025 (data minimization).

**Fix:**
1. In `NotiFetchListenerService.kt`, replace any call to `extrasToJson(extras)`
   with a hardcoded empty JSON object: `extrasJson = "{}"`
2. Do NOT remove the `extrasJson` field from the Room entity (it would require
   a schema migration — handle that in BUG 4). Just stop populating it.
3. You may optionally mark `extrasToJson()` in Helpers.kt as `@Deprecated`
   with a note explaining why it must not be called.

After this fix, the app only stores: title, text, bigText, subText — exactly
what is visible to the user in the notification shade. Nothing more.

### BUG 4: No Room migration strategy (data destruction on any schema change)
**Files:**
- `data/local/NotiFetchDatabase.kt`
- `di/DatabaseModule.kt` (line ~28)

**Problem:** `fallbackToDestructiveMigration()` is used. Any schema change
destroys all user data. The app is at Room schema v5.

**Fix — two parts:**

**Part A:** Write a real migration from v5 to v6. The v6 change should be to
mark `extrasJson` as deprecated but keep it in the schema (to preserve backward
compatibility with any existing data). Add a new `isBookmarked` Boolean column
(default false) to `captured_notifications` — this is needed for the bookmarks
feature later and is a clean reason to bump to v6.

```kotlin
val MIGRATION_5_6 = object : Migration(5, 6) {
    override fun migrate(database: SupportSQLiteDatabase) {
        // Add isBookmarked column, default false
        database.execSQL(
            "ALTER TABLE captured_notifications ADD COLUMN isBookmarked INTEGER NOT NULL DEFAULT 0"
        )
        // Clear extrasJson data for privacy compliance
        database.execSQL("UPDATE captured_notifications SET extrasJson = '{}'")
    }
}
```

**Part B:** Register the migration in the database builder:
```kotlin
Room.databaseBuilder(context, NotiFetchDatabase::class.java, "notifetch_database")
    .addMigrations(MIGRATION_5_6)
    // Keep fallback as LAST resort only, add a log warning before it triggers
    .fallbackToDestructiveMigration()
    .build()
```

Update `CapturedNotification.kt` entity: add `val isBookmarked: Boolean = false`
and bump the Room schema version to 6.

           android:value="conversations"/>

filter using `Constants.ALL_PACKAGES.contains(sbn.packageName)` runs correctly
and is the sole gating mechanism. No metadata fallback needed.

### BUG 7: initializePlatformConfigs race condition
**File:** `data/repository/NotificationRepository.kt` lines ~187-204

**Problem:**
1. The initialization check only verifies whether the FIRST package in the list
   has a config row. If the DB was partially initialized (app crash during init),
   all subsequent packages are missing but the check passes as "done".
2. This function is called from both `NotiFetchApp.onCreate()` AND
   `HomeViewModel.init`, creating a potential race condition.

**Fix:**
```kotlin
// Instead of checking only first package:
private suspend fun initializePlatformConfigs() {
    // Use a mutex to prevent concurrent initialization
    mutex.withLock {
        val existingCount = platformConfigDao.getConfigCount()
        if (existingCount >= Constants.ALL_PACKAGES.size) return

        // Initialize missing configs only (idempotent)
        Constants.ALL_PACKAGES.forEach { (packageName, displayName) ->
            val existing = platformConfigDao.getConfig(packageName)
            if (existing == null) {
                platformConfigDao.insert(
                    PlatformConfig(
                        packageName = packageName,
                        displayName = displayName,
                        isEnabled = true,
                        notificationCount = 0
                    )
                )
            }
        }
    }
}
```

Add a `private val mutex = Mutex()` to `NotificationRepository`. Remove the
call from `HomeViewModel.init` — only `NotiFetchApp.onCreate()` should call it.

suspend fun deleteAllDataIncludingServer(): Result<Unit> {

**Part B: Backend** — Add this API endpoint to the Next.js backend:
```typescript
// src/app/api/user/data/route.ts
import { NextRequest, NextResponse } from 'next/server'
import { verifyAuthToken } from '@/lib/auth'
import { prisma } from '@/lib/prisma'

export async function DELETE(request: NextRequest) {
    const userId = await verifyAuthToken(request)
    if (!userId) {
        return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    }
    await prisma.notification.deleteMany({ where: { userId } })
    await prisma.user.delete({ where: { id: userId } })
    return NextResponse.json({ success: true })
}
```

### BUG 9: EarningsViewModel frozen timestamps
**File:** `ui/viewmodel/EarningsViewModel.kt` lines ~43-63

**Problem:** Timestamps for "today", "this week", "this month" are captured
once at ViewModel construction. If the app stays open past midnight, the stats
show stale data.

**Fix:** Mirror the reactive timestamp approach used in HomeViewModel. Use
`StateFlow` values that update at midnight:

```kotlin
private val _currentDay = MutableStateFlow(getStartOfDay())
val currentDay: StateFlow<Long> = _currentDay.asStateFlow()

init {
    viewModelScope.launch {
        while (true) {
            val msUntilMidnight = getMillisUntilMidnight()
            delay(msUntilMidnight)
            _currentDay.value = getStartOfDay()
            _currentWeek.value = getStartOfWeek()
            _currentMonth.value = getStartOfMonth()
        }
    }
}

private fun getMillisUntilMidnight(): Long {
    val now = Calendar.getInstance()
    val midnight = Calendar.getInstance().apply {
        add(Calendar.DAY_OF_MONTH, 1)
        set(Calendar.HOUR_OF_DAY, 0)
        set(Calendar.MINUTE, 0)
        set(Calendar.SECOND, 0)
        set(Calendar.MILLISECOND, 0)
    }
    return midnight.timeInMillis - now.timeInMillis
}
```

`AndroidViewModel`) and inject context via `@ApplicationContext` through Hilt.

class WeeklySummaryWorker(ctx: Context, params: WorkerParameters) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result {
        val lastWeekEarnings = repository.getEarningsForLastWeek()
        val prevWeekEarnings = repository.getEarningsForPreviousWeek()
        val topPlatform = repository.getTopPlatformLastWeek()
        val change = if (prevWeekEarnings > 0)
            ((lastWeekEarnings - prevWeekEarnings) / prevWeekEarnings * 100).toInt()
        else 0

        val changeText = when {
            change > 0 -> "▲ $change% vs last week"
            change < 0 -> "▼ ${-change}% vs last week"
            else -> "Same as last week"
        }

        sendLocalNotification(
            title = "Your week in review",
            body = "₹${lastWeekEarnings.toInt()} earned · Top: $topPlatform · $changeText"
        )
        return Result.success()
    }
}

val deepLinkUri: String? = try {

### API 2: Aggregate platform stats (foundation for data moat features)
```typescript
// src/app/api/stats/platforms/route.ts
// Returns anonymized aggregate stats across all users
// Used by future benchmarking and platform intelligence features
export async function GET(request: NextRequest) {
    const city = request.nextUrl.searchParams.get('city')
    const platform = request.nextUrl.searchParams.get('platform')
    const period = request.nextUrl.searchParams.get('period') || '7d'

    // Aggregate query — no individual user data exposed
    const stats = await prisma.notification.groupBy({
        by: ['platform'],
        where: {
            receivedAt: { gte: getPeriodStart(period) },
            ...(city ? { city } : {}),
            ...(platform ? { platform } : {}),
        },
        _avg: { orderValue: true },
        _count: { id: true },
    })

    return NextResponse.json({
        stats,
        sampleSize: stats.reduce((sum, s) => sum + s._count.id, 0),
        period,
        // Never return data with sample size < 10 (privacy protection)
        isStatisticallyValid: stats.every(s => s._count.id >= 10)
    })
}
```
